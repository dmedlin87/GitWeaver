## 2026-03-01 - Redaction Regex Partial Leakage
**Vulnerability:** Updating a regex to strictly match a shorter length (e.g. `/npm_[a-zA-Z0-9]{32}/gi`) when longer legacy lengths exist (`36-40`) causes partial redaction, leaking the trailing characters into plaintext logs.
**Learning:** Redaction patterns must always use length ranges or unbounded ends to cover all valid variations, rather than enforcing strict exact lengths that might partially match and leak.
**Prevention:** Ensure test cases explicitly verify that standard and legacy lengths are completely redacted without partial matches, and utilize length ranges like `{32,40}` in redaction regex definitions.
## 2025-03-03 - Case-Insensitive Credential Leakage on Windows
**Vulnerability:** The orchestrator leaked host environment credentials (e.g. AWS_ACCESS_KEY_ID, OPENAI_API_KEY) to untrusted child processes executing inside the PTY when users configured credentials with non-standard casing (e.g. `aws_access_key_id`) on Windows platforms.
**Learning:** Node.js exposes Windows environment variables case-insensitively during direct lookups (`process.env.aws_access_key_id`), but retains their original case during enumeration (`...process.env`, `Object.keys()`). Using strict string-equality against a `DENYLIST` to filter `process.env` fails to scrub case-variations, silently passing the secret to the child process.
**Prevention:** Always use case-insensitive matching (`toLowerCase()`) when performing denylist scrubbing on environment objects to guarantee credentials cannot leak via casing variations. Apply this consistently across all execution modes (`pty-manager`, `host-executor`, `advisory-executor`) as a defense-in-depth measure.
