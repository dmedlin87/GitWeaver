# Architecture Drift Report - 2025-02-19

## Invariants Checked

1.  **Command Policy Enforcement**: "Commands generated by LLMs, particularly gateCommand in the TaskContract, must be validated against a commandPolicy".
    *   *Finding*: Missing enforcement in `src/core/orchestrator.ts`. The `gateCommand` was being executed without validation against the `TaskContract.commandPolicy`.
    *   *Fix*: Implemented `validateCommand` in `src/verification/command-policy.ts` and integrated it into `Orchestrator.executeTask` before `runGate`.
    *   *Verification*: Added `tests/unit/command-policy.test.ts` and `tests/unit/orchestrator-policy.test.ts` to prove enforcement.

## Gaps Found

*   **Gate Command Policy Bypass**: The `gateCommand` (whether explicitly defined in the task or falling back to the baseline default) was not being checked against the task's allow/deny list. This could allow an agent to specify a malicious gate command (if the planner was compromised or hallucinated) or allow a default command that violates a specific task's stricter policy.

## Fixes Applied

*   **New Module**: `src/verification/command-policy.ts` implementing `validateCommand` logic (allow-list prefix matching, deny-list inclusion check, deny-by-default if allow-list is empty).
*   **Orchestrator Integration**: Modified `src/core/orchestrator.ts` to import and call `validateCommand` before executing the gate command. Throws `ABORTED_POLICY` if validation fails.
*   **Tests**:
    *   Unit tests for `validateCommand` covering edge cases.
    *   Integration-style unit tests for `Orchestrator` mocking the execution flow to verify the policy check prevents execution.

## Remaining Risks

*   **Config Default Policy**: The global configuration default command policy is not currently merged with the task policy in the planner, but the task policy is authoritative during execution. If the planner generates an empty policy, nothing runs. This is safe (fail-closed) but might lead to usability issues if the planner doesn't populate `commandPolicy` correctly.
*   **Shell Injection in Arguments**: The validation uses simple string matching (`startsWith`, `includes`). Sophisticated shell injection (e.g. `pnpm test; rm -rf /`) is caught if the deny list includes `;` or `&`, or if the allowed prefix is strict enough. The current `config.ts` default deny list should be robust, but we rely on the planner to set safe policies.
